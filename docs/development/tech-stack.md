## 1. 技术选型总览

### 1.1 核心决策

| 技术领域 | 选型方案 | 决策理由 |
|----------|----------|----------|
| **分发渠道** | 微信小程序 | 无需下载、分享方便、跨平台 |
| **前端框架** | 微信小程序原生 + TypeScript | 生态成熟、性能优化、类型安全 |
| **P2P 通信** | WebRTC | 标准方案、低延迟、无需服务器中转 |
| **信令服务** | Node.js + WebSocket | 轻量、仅用于匹配、不存储数据 |
| **状态管理** | 本地存储 + 内存 | 单机模式无需后端、P2P 房主托管 |
| **AI 引擎** | 自研规则引擎 | 可控、无需外部依赖、响应快 |

## 2. 前端技术栈

### 2.1 微信小程序

**基础库版本**: ≥ 2.19.0

**选型理由**:
- ✅ 无需下载安装，符合碎片化场景
- ✅ 分享方便（邀请码一键分享）
- ✅ 跨平台（iOS/Android 通吃）
- ✅ 性能可接受（掼蛋游戏不复杂）
- ✅ 开发成本低（生态成熟）

**技术约束**:
- 包体大小限制：主包 ≤ 2MB，分包 ≤ 20MB
- 内存限制：≤ 512MB（建议控制在 200MB 内）
- 网络请求：需备案域名（信令服务器）

### 2.2 开发语言：TypeScript

**选型理由**:
- ✅ 类型安全，减少运行时错误
- ✅ 代码可维护性好
- ✅ 微信小程序官方支持
- ✅ 团队熟悉度高

**配置**:
```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

### 2.3 状态管理：本地存储

**方案**: `wx.setStorageSync` + 内存状态

**选型理由**:
- ✅ 单机模式无需后端
- ✅ 暂停/继续功能依赖本地持久化
- ✅ 简单可靠，无额外依赖

**数据结构**:
```typescript
interface GameState {
  mode: 'single' | 'p2p';
  players: Player[];
  hands: Card[][];
  playedCards: Card[];
  currentTurn: number;
  lastMove: Move | null;
  paused: boolean;
  timestamp: number;
}
```

### 2.4 UI 渲染

**方案**: 微信小程序原生组件 + Canvas

**选型理由**:
- ✅ 原生组件性能好（60 FPS）
- ✅ Canvas 用于牌面绘制（灵活）
- ✅ 符合"简洁、不花哨"原则

**核心组件**:
- `view` / `text` - 基础布局
- `canvas` - 牌面绘制
- `button` - 交互按钮
- `image` - 素材资源

## 3. 后端技术栈

### 3.1 信令服务器（P2P 模式）

**技术选型**: Node.js + WebSocket

**选型理由**:
- ✅ 轻量级，仅用于匹配
- ✅ 不存储用户数据（符合隐私要求）
- ✅ 开发成本低
- ✅ 可部署在低成本云函数

**技术栈**:
| 组件 | 选型 | 说明 |
|------|------|------|
| **运行时** | Node.js 18+ | 成熟稳定 |
| **WebSocket** | `ws` 库 | 轻量高效 |
| **部署** | 云函数/容器 | 按需扩缩容 |
| **域名** | 需备案 | 微信小程序要求 |

**核心功能**:
```typescript
// 信令服务器伪代码
interface SignalingServer {
  // 创建房间，返回房间 ID 和邀请码
  createRoom(): Promise<{ roomId: string; inviteCode: string }>;
  
  // 加入房间
  joinRoom(inviteCode: string): Promise<{ roomId: string; players: Player[] }>;
  
  // 交换 WebRTC 信令（SDP/ICE）
  exchangeSignal(roomId: string, signal: any): void;
  
  // 房间心跳（检测断线）
  heartbeat(roomId: string): void;
}
```

**非功能要求**:
- 响应时间：≤ 100ms
- 并发能力：≥ 1000 房间（初期）
- 可用性：≥ 99%
- 成本：≤ ¥100/月（初期）

### 3.2 无需后端功能

**单机模式**: 完全本地运行，无需服务器

**理由**:
- ✅ 降低运营成本
- ✅ 用户隐私保护
- ✅ 无网络也可游戏
- ✅ 符合"无负担"产品定位

## 4. P2P 通信技术栈

### 4.1 WebRTC

**选型理由**:
- ✅ 标准方案，浏览器/小程序支持
- ✅ 低延迟（直接 P2P 通信）
- ✅ 加密传输（DTLS/SRTP）
- ✅ 无需服务器中转数据

**微信小程序支持**:
- `rtcpusher` / `rtcvplayer` 组件
- 需用户授权（摄像头/麦克风权限，但掼蛋不需要）
- 基础库 ≥ 2.19.0

**技术架构**:
```
┌─────────────┐         ┌─────────────┐
│   房主设备   │◄───────►│   玩家设备   │
│  (WebRTC)   │  P2P    │  (WebRTC)   │
└─────────────┘         └─────────────┘
       ▲                       ▲
       │                       │
       └───────────┬───────────┘
                   │
          ┌────────▼────────┐
          │  信令服务器     │
          │  (WebSocket)    │
          └─────────────────┘
```

### 4.2 信令协议

**协议格式**:
```json
{
  "type": "offer" | "answer" | "ice-candidate",
  "roomId": "room-123",
  "from": "player-1",
  "to": "player-2",
  "payload": { /* SDP or ICE */ }
}
```

**流程**:
1. 房主创建房间 → 信令服务器返回 roomId
2. 玩家通过邀请码加入 → 信令服务器匹配
3. 房主发送 WebRTC Offer → 信令服务器转发
4. 玩家回复 WebRTC Answer → 信令服务器转发
5. 交换 ICE Candidate → 建立 P2P 连接
6. 连接建立后，信令服务器不再参与

## 5. AI 引擎技术栈

### 5.1 自研规则引擎

**选型理由**:
- ✅ 完全可控（出牌策略可调）
- ✅ 无需外部依赖（离线可用）
- ✅ 响应快（≤ 300ms）
- ✅ 符合"像真人"要求

**技术实现**:
```typescript
interface AIEngine {
  // 分析当前牌局
  analyze(gameState: GameState): Analysis;
  
  // 选择出牌
  selectMove(hand: Card[], analysis: Analysis): Move;
  
  // 记忆已出牌
  remember(playedCards: Card[]): void;
  
  // 模拟思考时间（100-300ms 随机）
  think(): Promise<void>;
}
```

**核心模块**:
| 模块 | 功能 | 复杂度 |
|------|------|--------|
| **牌型识别** | 识别单张/对子/顺子等 | 中 |
| **牌力评估** | 评估手牌强度 | 中 |
| **策略选择** | 根据牌局选择出牌 | 高 |
| **记忆系统** | 记住已出大牌 | 低 |

## 7. 技术栈对比

### 7.1 前端框架对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **微信小程序原生** | 性能最佳、生态成熟 | 仅限微信生态 | ⭐⭐⭐⭐⭐ |
| **uni-app** | 跨平台（多端） | 性能略差、学习成本 | ⭐⭐⭐ |
| **Taro** | 跨平台、React 语法 | 性能略差、调试复杂 | ⭐⭐⭐ |

**决策**: 微信小程序原生（符合产品定位）

### 7.2 P2P 方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **WebRTC** | 标准、低延迟、加密 | 需要信令服务器 | ⭐⭐⭐⭐⭐ |
| **第三方 P2P** | 简单易用 | 依赖第三方、成本 | ⭐⭐⭐ |
| **局域网** | 无需服务器 | 距离限制、不实用 | ⭐⭐ |

**决策**: WebRTC + 轻量信令服务器

## 8. 技术约束与合规

### 8.1 微信小程序约束

- **包体大小**: 主包 ≤ 2MB，总分包 ≤ 20MB
- **网络请求**: 需备案域名（HTTPS）
- **用户隐私**: 需隐私协议（无数据收集可简化）
- **审核**: 游戏类目需版号（单机休闲可规避）

### 8.2 合规建议

- ✅ 不收集用户隐私数据
- ✅ 不依赖虚拟货币/充值
- ✅ 单机模式无需版号
- ⚠️ P2P 联机可能需版号（建议初期仅单机）

## 9. 技术栈总结

### 9.1 核心选型

```
┌─────────────────────────────────────────────┐
│              掼蛋大师 v3 技术栈              │
├─────────────────────────────────────────────┤
│  前端：微信小程序原生 + TypeScript          │
│  后端：无（单机） / Node.js 信令（P2P）     │
│  通信：WebRTC（P2P） + WebSocket（信令）    │
│  存储：本地存储（wx.setStorageSync）        │
│  AI：自研规则引擎                           │
└─────────────────────────────────────────────┘
```

### 9.2 选型原则

1. **简单优先** - 不引入不必要的复杂度
2. **本地优先** - 单机模式无需后端
3. **性能优先** - 响应时间 ≤ 300ms
4. **成本优先** - 运营成本 ≤ ¥100/月
5. **隐私优先** - 不收集用户数据
