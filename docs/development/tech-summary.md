## 2. 核心技术决策

### 2.1 分发渠道：微信小程序

**决策**: ✅ 微信小程序优先

**理由**:
- 无需下载安装，符合碎片化场景
- 分享方便（邀请码一键分享）
- 跨平台（iOS/Android 通吃）
- 开发成本低（生态成熟）
- 性能可接受（掼蛋游戏不复杂）

**技术约束**:
- 包体大小：主包 ≤ 2MB，总分包 ≤ 20MB
- 内存限制：≤ 512MB（建议控制在 200MB 内）
- 网络请求：需备案域名（HTTPS）
- 基础库版本：≥ 2.19.0（WebRTC 支持）

**对比分析**:

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **微信小程序** | 无需下载、分享方便、跨平台 | 性能受限、包体限制 | ⭐⭐⭐⭐⭐ |
| **原生 App** | 性能最佳、功能完整 | 需要下载、分发困难 | ⭐⭐⭐ |
| **H5** | 无需下载、跨平台 | 性能较差、体验一般 | ⭐⭐ |

### 2.2 前端框架：小程序原生 + TypeScript

**决策**: ✅ 微信小程序原生框架 + TypeScript

**理由**:
- 性能最佳（原生组件）
- 生态成熟（官方文档、社区支持）
- 类型安全（TypeScript 减少运行时错误）
- 开发效率高（微信开发者工具）

**技术栈**:
```
微信小程序
├── 语言：TypeScript 4.x
├── 基础库：≥ 2.19.0
├── UI：原生组件 + Canvas
├── 状态管理：本地存储 + 内存
├── 构建工具：微信开发者工具
└── 代码规范：ESLint + Prettier
```

### 2.3 P2P 通信：WebRTC

**决策**: ✅ WebRTC DataChannel

**理由**:
- 标准方案（W3C 标准，浏览器/小程序支持）
- 低延迟（P2P 直连，延迟 ≤ 50ms）
- 加密传输（DTLS/SRTP 内置）
- 无需服务器中转数据（降低成本）
- 支持断线重连

**技术架构**:
```
房主 ←───WebRTC DataChannel───→ 玩家
  ▲                               ▲
  │                               │
  └──── WebSocket (信令) ────────┘
              ↓
       信令服务器
   （仅用于初始匹配）
```

**对比分析**:

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **WebRTC** | 标准、低延迟、加密、免费 STUN | 需要信令服务器、NAT 穿透可能失败 | ⭐⭐⭐⭐⭐ |
| **第三方 P2P** | 简单易用、NAT 穿透好 | 依赖第三方、可能收费 | ⭐⭐⭐ |
| **WebSocket 中转** | 简单可靠 | 服务器成本高、延迟高 | ⭐⭐ |

### 2.4 信令服务：Node.js + WebSocket

**决策**: ✅ Node.js + ws 库 + 云函数部署

**理由**:
- 轻量级（仅用于匹配，不存储数据）
- 开发成本低（JavaScript 技术栈）
- 部署灵活（云函数/容器）
- 成本低廉（初期 ¥0-50/月）

**核心功能**:
- 创建房间（生成邀请码）
- 加入房间（验证邀请码）
- 信令转发（Offer/Answer/ICE）
- 心跳检测（5 分钟过期清理）

**非功能要求**:
- 响应时间：≤ 100ms
- 并发能力：≥ 1000 房间
- 可用性：≥ 99%
- 成本：≤ ¥100/月

### 2.5 存储方案：本地存储

**决策**: ✅ wx.setStorageSync（单机模式）

**理由**:
- 单机模式无需后端
- 暂停/继续功能依赖本地持久化
- 简单可靠，无额外依赖
- 隐私保护（数据不出设备）

**存储内容**:
- 游戏状态（牌局、手牌、出牌记录）
- 用户设置（音效、难度）
- 游戏历史（最近 30 条）

### 2.6 AI 引擎：自研规则引擎

**决策**: ✅ 自研规则引擎（非机器学习）

**理由**:
- 完全可控（出牌策略可调）
- 无需外部依赖（离线可用）
- 响应快（≤ 300ms）
- 符合"像真人"要求（可模拟思考时间）

**核心模块**:
- 牌型识别（单张/对子/顺子/炸弹等）
- 牌力评估（手牌强度计算）
- 策略选择（根据牌局选择出牌）
- 记忆系统（记住已出大牌）

## 3. 技术风险评估

### 3.1 高风险项

| 风险 | 可能性 | 影响 | 应对措施 | 负责人 |
|------|--------|------|----------|--------|
| **NAT 穿透失败** | 中 (30%) | 高 | 备用 TURN 服务器、提示切换网络 | 开发 |
| **微信小程序审核** | 中 (40%) | 高 | 单机模式无需版号、P2P 模式暂缓 | PM |
| **iOS 后台限制** | 高 (70%) | 中 | 提示用户不要切出、本地保存状态 | 开发 |

### 3.2 中风险项

| 风险 | 可能性 | 影响 | 应对措施 | 负责人 |
|------|--------|------|----------|--------|
| **信令服务器宕机** | 低 (10%) | 高 | 多实例部署、自动重启、监控告警 | 运维 |
| **P2P 断线重连失败** | 中 (30%) | 中 | 优化重连逻辑、明确用户提示 | 开发 |
| **AI 出牌太机械** | 中 (40%) | 中 | 增加随机性、模拟真人思考时间 | 开发 |

### 3.3 低风险项

| 风险 | 可能性 | 影响 | 应对措施 | 负责人 |
|------|--------|------|----------|--------|
