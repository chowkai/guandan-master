# 掼蛋项目 - 全流程管理规范 v2.0

**版本**: v2.0  
**创建日期**: 2026-03-25  
**修订原因**: Phase 3 出现 4 个重复性问题（牌间距、事件绑定、首发提示、流程漏洞）  
**维护人**: main (PM)  
**生效日期**: 立即生效

---

## 📋 问题回顾（Phase 3 教训）

### 重复出现的问题

| 问题 | 出现次数 | 根本原因 |
|------|----------|----------|
| 牌间距太近 | 2 次 | 需求未量化验收标准 |
| onCardTap 事件缺失 | 1 次 | 设计→开发信息传递丢失 |
| 首发提示缺失 | 2 次 | UI 设计遗漏需求 |
| BUG 反复出现 | 多次 | 修复后无验证闭环 |

### 核心问题

1. **需求未量化** - "牌间距合适" → 应为"牌间距≥16px，触摸区域≥44×44pt"
2. **信息传递丢失** - 设计文档未同步给开发
3. **验收缺失** - 修复后无截图/录屏确认
4. **真机测试缺失** - 仅在模拟器测试

---

## 🎯 全流程概览

```
需求阶段 → 设计阶段 → 开发阶段 → 测试阶段 → 运维阶段
   ↓          ↓          ↓          ↓          ↓
 评审       评审       代码       验收       监控
 签字       签字       审查       测试       反馈
```

**核心原则**:
1. **每个阶段必须有输出物**
2. **每个输出物必须有评审签字**
3. **每个 BUG 必须有验证闭环**
4. **所有信息必须留存可追溯**

### 输出物（必须）

| 文档 | 内容要求 | 负责人 | 评审人 |
|------|----------|--------|--------|
| `requirements/product-requirements.md` | 功能清单 + 用户故事 | PM | 用户 |
| `requirements/acceptance-criteria.md` | **量化验收标准** | PM | 开发 + 测试 |
| `requirements/technical-constraints.md` | 技术限制 + 风险 | 开发 | PM |

### 验收标准模板（关键改进）

```markdown
## 功能 X: 手牌选择

### 功能性验收标准
- [ ] 点击卡牌可选中/反选
- [ ] 选中后卡牌向上移动 10-15px
- [ ] 最多可选 5 张牌

### 非功能性验收标准（量化）
- [ ] 牌间距 ≥ 16px（确保手指触摸不误触）
- [ ] 单张牌触摸区域 ≥ 44×44pt（iOS 人机界面指南）
- [ ] 响应时间 < 100ms
- [ ] 支持设备：iPhone SE ~ iPhone 15 Pro Max

### 验收方法
- [ ] 真机测试（iPhone 12 及以上）
- [ ] 录屏验证
- [ ] 用户确认签字
```

### 评审检查清单

- [ ] 所有功能有量化验收标准
- [ ] 所有非功能需求有具体数值
- [ ] 技术风险已识别
- [ ] 开发负责人已评审签字
- [ ] 测试负责人已评审签字

| `docs/design/design-review-record.md` | **设计评审记录** | PM | 全体 |

### UI 设计稿要求（关键改进）

```markdown
## 页面：游戏主界面

### 布局标注
- 手牌区域高度：180px
- 牌间距：16px（小屏设备 12px）
- 卡牌尺寸：80×120px
- 选中偏移：向上 12px

### 状态说明
- 默认状态：卡牌水平排列
- 选中状态：卡牌向上偏移 12px，添加阴影
- 禁用状态：透明度 50%

### 响应式规则
- 屏幕宽度 < 375px：牌间距 12px
- 屏幕宽度 ≥ 375px：牌间距 16px
- 屏幕宽度 ≥ 768px（平板）：牌间距 20px

### 信息提示区
- 位置：顶部，高度 50px
- 内容："当前出牌：玩家 X" / "等待 XX 出牌..."
- 首发提示：高亮显示"玩家 X 首发"
```

### 设计评审检查清单

- [ ] 所有需求已体现在设计稿中
- [ ] 所有交互状态有说明
- [ ] 响应式规则明确
- [ ] **开发确认可实现**
- [ ] **测试确认有验收标准**
- [ ] 评审记录已签字

### 输出物（必须）

| 文档 | 内容要求 | 负责人 | 评审人 |
|------|----------|--------|--------|
| `docs/development/implementation-plan.md` | 实现方案 + 技术选型 | 开发 | PM |
| `src/` | 源代码 | 开发 | - |
| `docs/development/code-review-record.md` | **代码审查记录** | 开发 | PM/测试 |
| `docs/development/self-test-record.md` | **自测记录 + 截图** | 开发 | 测试 |

### 开发流程（关键改进）

```
1. 阅读需求 + 设计文档（30 分钟）
   ↓
2. 确认理解无误（有疑问立即提出）
   ↓
3. 编写实现方案（复杂功能必须）
   ↓
4. 编码实现
   ↓
5. **自测 + 截图/录屏**（必须真机或模拟器）
   ↓
6. 提交代码审查
   ↓
7. 根据审查意见修改
   ↓
8. 提交测试
```

## 🧪 阶段 4: 测试阶段

### 输入
- 源代码
- 自测记录
- 验收标准

### 输出物（必须）

| 文档 | 内容要求 | 负责人 | 评审人 |
|------|----------|--------|--------|
| `docs/testing/test-cases.md` | 测试用例 | 测试 | PM |
| `docs/testing/test-report.md` | **测试报告 + 截图** | 测试 | PM + 开发 |
| `docs/testing/bug-report.md` | BUG 清单（如有） | 测试 | 开发 |
| `docs/testing/acceptance-record.md` | **验收签字** | PM | 用户 |

### 测试流程（关键改进）

```
1. 阅读验收标准 + 自测记录
   ↓
2. 编写测试用例
   ↓
3. **真机测试**（必须，不能仅模拟器）
   ↓
4. 记录测试结果 + 截图/录屏
   ↓
5. 如有 BUG，提交 BUG 报告
   ↓
6. 开发修复后，**回归测试**
   ↓
7. 验收签字
```

| 真机触摸 | 不误触 | ✅ | 通过 |

### 回归测试要求（关键改进）

**BUG 修复后必须**:
1. 开发提供修复截图/录屏
2. 测试**真机回归验证**
3. 用户确认签字后方可关闭 BUG

### 阶段完成标志
✅ 测试报告 + 验收签字 + BUG 全部关闭

## 🔧 阶段 5: 运维阶段（BUG 修复与体验优化）

### 输入
- 用户反馈
- 监控数据
- BUG 报告

### 输出物（必须）

| 文档 | 内容要求 | 负责人 | 评审人 |
|------|----------|--------|--------|
| `tasks/bug-report-{id}.md` | BUG 详情 + 复现步骤 | 发现人 | PM |
| `tasks/fix-plan-{id}.md` | 修复方案 + 影响评估 | 开发 | PM |
| `tasks/fix-verification-{id}.md` | **修复验证 + 截图** | 测试 | PM |
| `docs/operations/incident-report.md` | 事故报告（P0/P1） | PM | 全体 |

### BUG 修复流程（关键改进）

```
1. 发现 BUG
   ↓
2. 记录复现步骤 + 截图/录屏
   ↓
3. 确定优先级（P0-P3）
   ↓
4. 分配修复人
   ↓
5. 开发修复 + **自测截图**
   ↓
6. 测试**真机回归**
   ↓
7. **用户确认签字**
   ↓
8. 关闭 BUG
```

```
1. 收集优化建议
   ↓

   ↓

```

### 版本控制

- 所有文档必须 Git 提交
- 提交信息格式：`type: 描述`
- 重要文档变更必须 PR 审查

### 追溯链

```
用户需求 → 需求文档 → 设计稿 → 代码 → 测试报告 → 验收签字
   ↓          ↓          ↓       ↓         ↓          ↓
 签字       签字       签字    审查      签字      签字
```

**任何环节缺失，不得进入下一阶段**
