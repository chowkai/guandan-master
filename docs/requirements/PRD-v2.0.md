## 1. 背景与目标

### 1.1 背景
- 掼蛋在江苏/安徽/山东等地广泛流行，是社交娱乐的重要方式
- 微信小程序是最佳载体（无需下载、社交传播快）
- 现有竞品存在体验问题（出牌卡顿、规则不全、界面粗糙）

### 1.2 目标
- **业务目标**: 上线 3 个月内 DAU 达到 1 万，次日留存≥40%
- **用户目标**: 随时随地约朋友打掼蛋，操作流畅，规则准确

### 1.3 范围
**包含**:
- 微信小程序（前端）
- 信令服务器（后端）
- 用户系统、战绩系统
- 完整掼蛋规则实现

**不包含**:
- APP（iOS/Android）
- 支付系统（Phase 2）
- 赛事系统（Phase 2）

---

## 2. 用户研究

### 2.1 目标用户
- **核心用户**: 25-45 岁，江苏/安徽/山东地区，有掼蛋基础
- **次要用户**: 18-24 岁，大学生，学习掼蛋规则
- **使用场景**: 朋友聚会、通勤路上、休息时间

### 2.2 用户故事
```
As a 掼蛋爱好者，I want 随时约朋友打掼蛋，So that 不用凑到一起也能玩
As a 新手，I want 有详细的规则说明和提示，So that 快速学会掼蛋
As a 老手，I want 流畅的出牌体验和准确的判牌，So that 享受游戏乐趣
As a 社交达人，I want 邀请好友一起玩游戏，So that 增进感情
```

### 2.3 竞品分析
| 竞品 | 优点 | 缺点 | 我们的差异化 |
|------|------|------|-------------|
| 掼蛋之家 | 规则齐全 | 界面老旧、广告多 | 简洁界面、无广告 |
| 腾讯掼蛋 | 用户量大 | 匹配慢、氪金重 | 好友约战、公平竞技 |
| 掼蛋大师 v1 | 本地化好 | 功能单一 | 社交系统、战绩记录 |

### 3.2 详细需求

#### F1. 登录/注册

**描述**:
用户使用微信授权登录，自动注册账号。

**用户流程**:
```
打开小程序 → 授权登录 → 获取用户信息 → 注册/登录 → 进入首页
```

**界面要求**:
- 登录页：微信授权按钮、用户协议
- 首页：顶部显示用户头像、昵称

**数据要求**:
- 用户表：openid、nickname、avatar、create_time

**验收标准**:
- [ ] 微信授权成功，获取用户信息
- [ ] 新用户自动注册，老用户直接登录
- [ ] 登录失败有明确提示

**边界情况**:
- 网络异常：提示"网络异常，请重试"
- 拒绝授权：引导用户开启授权

---

#### F2. 创建房间

**描述**:
用户创建掼蛋房间，可自定义游戏规则。

**用户流程**:
```
点击"创建房间" → 选择游戏规则 → 确认创建 → 生成房间号 → 等待好友加入
```

**界面要求**:
- 创建房间页：规则选项（打 2、进贡、局数）
- 房间页：房间号、玩家列表、开始按钮

**数据要求**:
- 房间表：room_id、creator_id、rules、status、create_time

**验收标准**:
- [ ] 可自定义规则（打 2/打 3、进贡/不进贡）
- [ ] 生成 6 位房间号
- [ ] 房间满 4 人可开始游戏

**边界情况**:
- 房间已满：提示"房间已满"
- 重复创建：提示"已有进行中的房间"

---

#### F3. 加入房间

**描述**:
用户通过房间号或邀请加入已有房间。

**用户流程**:
```
点击"加入房间" → 输入房间号 → 确认加入 → 进入房间等待
```

**界面要求**:
- 加入房间页：房间号输入框、确认按钮
- 房间页：房间号、玩家列表、准备按钮

**数据要求**:
- 房间玩家表：room_id、user_id、seat、ready

**验收标准**:
- [ ] 输入房间号可加入
- [ ] 好友邀请可直接加入
- [ ] 房间不存在/已满有明确提示

**边界情况**:
- 房间不存在：提示"房间不存在"
- 房间已满：提示"房间已满"
- 已在房间中：提示"已有进行中的房间"

---

#### F4. 发牌

**描述**:
游戏开始后，随机分发牌并展示给各玩家。

**用户流程**:
```
房主点击"开始" → 洗牌动画 → 发牌动画 → 玩家获得手牌
```

**界面要求**:
- 发牌动画：牌从中间向四周移动
- 手牌区：玩家手牌横向排列

**数据要求**:
- 牌堆：54 张牌（2 副）
- 玩家手牌：27 张/人

**验收标准**:
- [ ] 发牌动画流畅（60fps）
- [ ] 手牌正确展示（27 张）
- [ ] 牌面清晰可辨

**边界情况**:
- 网络异常：重连后恢复手牌

---

#### F5. 出牌

**描述**:
玩家选择手牌并出牌，支持提示、托管等功能。

**用户流程**:
```
轮到自己 → 选择手牌 → 点击"出牌" → 牌型验证 → 出牌成功
```

**界面要求**:
- 手牌区：可点击选牌、滑动排序
- 操作区：出牌、提示、托管、不出按钮
- 出牌区：展示其他玩家出的牌

**数据要求**:
- 出牌记录：user_id、cards、type、timestamp

**验收标准**:
- [ ] 选牌准确（点击/滑动）
- [ ] 牌型验证正确
- [ ] 提示功能准确推荐可出的牌
- [ ] 托管自动出最小牌

**边界情况**:
- 超时未出：自动托管
- 牌型错误：提示"牌型不正确"

---

#### F6. 牌型判断

**描述**:
判断出牌是否符合规则，比较牌型大小。

**牌型支持**:
- 单张、对子、三张、三带二、顺子、连对、钢板、炸弹、同花顺

**验收标准**:
- [ ] 所有牌型判断准确
- [ ] 牌型大小比较正确
- [ ] 支持掼蛋特殊规则（逢人配、进贡）

**边界情况**:
- 多人出相同牌型：判断正确大小
- 炸弹与普通牌型：炸弹优先

---

#### F7. 游戏结算

**描述**:
一局游戏结束后，计分、升级、显示结果。

**用户流程**:
```
一方出完牌 → 游戏结束 → 显示结果 → 计分 → 升级 → 准备下一局
```

**界面要求**:
- 结算页：胜负结果、得分、升级情况
- 统计：本局出牌数、炸弹数

**数据要求**:
- 战绩表：user_id、win/lose、score、update_time

**验收标准**:
- [ ] 胜负判断正确
- [ ] 计分准确（头游、二游、三游、末游）
- [ ] 升级正确（打 2→打 3→...→打 A）

**边界情况**:
- 中途退出：判负，对方获胜

## 4. 非功能需求

### 4.1 性能要求
- 首屏加载：< 3s
- 发牌动画：60fps
- 出牌响应：< 500ms
- WebSocket 延迟：< 200ms

### 4.2 兼容性要求
- 微信小程序：基础库 2.19.0+
- iOS: 12.0+
- Android: 8.0+

### 4.3 安全要求
- 用户数据加密存储
- WebSocket 鉴权
- 防作弊（牌堆随机性验证）

### 4.4 数据埋点
| 事件名 | 触发时机 | 上报字段 |
|--------|----------|----------|
| login | 登录成功 | user_id, timestamp |
| create_room | 创建房间 | room_id, rules |
| join_room | 加入房间 | room_id, user_id |
| play_card | 出牌 | user_id, cards, type |
| game_end | 游戏结束 | room_id, winner, score |
