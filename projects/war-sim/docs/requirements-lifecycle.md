**版本**: v0.1 (需求原型)  
**创建日期**: 2026-03-20  
**状态**: 需求论证阶段

---

## 📋 核心理念

**战争是由无数微小个体的决策构成的**。每个士兵不是冷冰冰的数字，而是有生存本能、有归属感、有晋升欲望的"人"。

**核心思想**：
- ✅ 每个单位都是独立个体，有自己的生命周期
- ✅ 个体行为遵循优先级原则（生存 > 编制 > 晋升）
- ✅ 玩家不能违背"人性"（不能随意牺牲个体）
- ✅ 宏观战争结局由微观个体决策汇聚而成

## 🧠 个体行为优先级

### 第一原则：确保生存

**生存威胁因素**：
| 威胁 | 描述 | 应对行为 |
|------|------|---------|
| **饥渴** | 饮食耗尽 | 寻找补给/投降/逃跑 |
| **伤重** | 生命值过低 | 寻求医疗/撤退 |
| **敌人袭击** | 遭遇敌方部队 | 战斗/逃跑/投降 |
| **力竭** | 体力耗尽 | 休息/求救 |

**恐慌机制**：
```
恐慌度 = f(饮食剩余，距离补给点，敌方数量，己方数量，士气)

恐慌度阈值：
- 30%: 开始焦虑，效率下降
- 60%: 士气快速下降
- 80%: 考虑逃跑/投降
- 100%: 强制溃散（特殊技能可减免）
```

**特殊技能豁免**：
| 技能 | 效果 | 适用兵种 |
|------|------|---------|
| **识途** | 远距离定位目标，不会迷路 | 斥候、传令兵 |
| **护身** | 减免恐慌度积累 | 斥候、传令兵 |
| **坚韧** | 提高恐慌阈值 | 老兵、军官 |

---

### 第二原则：依附编制

**编制的作用**：
1. **补给来源**：只有编制内才能获得稳定补给
2. **集群威慑**：借助集体力量对抗敌人
3. **命令传达**：接收上级指令，协同行动
4. **归属感**：降低恐慌度，提高士气

**溃散机制**：
```
当个体逃离编制超过视距/声距范围：

1. 失去补给来源 → 饮食消耗加快
2. 失去集群保护 → 遭遇敌人必败
3. 失去方向感 → 随机移动（无识途技能）
4. 士气持续下降 → 最终崩溃

可能的结局：
- 遇上敌方 → 俘虏/阵亡
- 遇上己方补给点 → 收容为散兵
- 体力耗尽 → 死亡
- 一段时间后 → 失踪（从历史中消失）
```

---

### 第三原则：渴望晋升

**晋升机制**：
```
晋升条件 = f(能力值，军功，资历，岗位空缺)

自动晋升：
- 系统根据能力与军功自动提拔
- 填补编制内空缺岗位

玩家干预：
- 玩家可以手动任命/降职
- 但过度干预会影响士气
```

**军功获取**：
| 行为 | 军功值 |
|------|--------|
| 击毙敌人 | +10 |
| 击伤敌人 | +5 |
| 完成任务 | +20 |
| 立功表现 | +15 |
| 逃跑/投降 | -50 |
| 被俘 | -30 |

## 🏗️ 技术可行性分析

### 核心挑战

**问题**：成千上万个个体，每个都需要实时做出决策，性能是否可行？

### 方案对比

| 方案 | 单位数 | 更新频率 | 性能 | 可行性 |
|------|--------|---------|------|--------|
| **每帧更新所有单位** | 10 万 | 60FPS | ❌ 不可行 | - |
| **时间分片更新** | 10 万 | 60FPS | ✅ 可行 | ⭐⭐⭐⭐ |
| **LOD 决策系统** | 10 万 | 60FPS | ✅ 可行 | ⭐⭐⭐⭐⭐ |
| **ECS 架构** | 10 万 | 60FPS | ✅ 可行 | ⭐⭐⭐⭐ |

### 推荐方案：LOD + 时间分片 + ECS

#### 1. LOD 决策系统（Level of Detail）

```
根据距离和重要性，采用不同更新频率：

LOD0 (重要单位 - 玩家附近/关键单位):
  - 每帧更新 (60Hz)
  - 完整 AI 决策
  - 精细动画
  
LOD1 (普通单位 - 可见范围):
  - 每秒 10 次 (10Hz)
  - 简化 AI 决策
  - 基础动画
  
LOD2 (远处单位 - 不可见但附近):
  - 每秒 1 次 (1Hz)
  - 极简决策
  - 无动画
  
LOD3 (超远单位 - 战略层面):
  - 每分钟 1 次 (1/60Hz)
  - 统计模拟
  - 无个体渲染
```

#### 2. 时间分片（Time Slicing）

```python
# 每帧只处理部分单位
def update_units(frame):
    units_per_frame = 1000  # 每帧处理 1000 个单位
    total_units = len(all_units)
    
    start_index = (frame % total_units)
    end_index = min(start_index + units_per_frame, total_units)
    
    for i in range(start_index, end_index):
        all_units[i].update()
```

#### 3. ECS 架构（Entity Component System）

```
优势：
- 数据与逻辑分离
- 缓存友好
- 易于并行处理
- 支持海量单位

结构：
Entity (单位 ID)
  ├── Components (数据)
  │   ├── PositionComponent
  │   ├── HealthComponent
  │   ├── AIComponent
  │   └── ...
  └── Systems (逻辑)
      ├── MovementSystem
      ├── CombatSystem
      ├── AISystem
      └── ...
```

## 🎮 游戏设计影响

### 玩家体验

**正面影响**：
- ✅ 每个士兵都是"活生生的人"，增强代入感
- ✅ 战术决策更真实，不能随意牺牲
- ✅ 每个单位有名字有履历，增强情感连接
- ✅ 战损不再是数字，而是真实的生命

**挑战**：
- ⚠️ 玩家自由度受限（不能违背人性）
- ⚠️ 需要更精细的微操管理
- ⚠️ 后勤压力更大（需要保障每个士兵的生存）

### 游戏平衡

**关键平衡点**：
1. **恐慌阈值**：太高=无敌士气，太低=一触即溃
2. **补给依赖**：太强=后勤决定一切，太弱=无需后勤
3. **晋升速度**：太快= unrealistic，太慢=无成就感
