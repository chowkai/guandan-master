## 🔄 标准工作流程

### 需求阶段

```
1. 需求收集 (PM + 周凯)
   ↓
2. 需求分析 (产品经理)
   ↓
3. 原型设计 (产品经理) ← 周凯确认
   ↓
4. UI 设计 (UI 设计师) ← PM + 周凯确认
   ↓
5. PRD 文档 (产品经理) ← PM + 开发 + 测试评审
```

### 开发阶段

```
6. 任务分解 (PM) ← 开发确认
   ↓
7. 技术设计 (架构师) ← PM + 开发评审
   ↓
8. 编码实现 (开发) ← Code Review 通过
   ↓
9. 测试验证 (测试) ← 100% 用例通过
   ↓
10. 用户验收 (周凯) ← 签字确认
```

## 🛡️ 质量保障体系

### Code Review Checklist

**代码规范**:
- [ ] 命名规范（变量/函数/类）
- [ ] 注释完整（复杂逻辑必有注释）
- [ ] 代码格式化（使用 linter）
- [ ] 无重复代码（DRY 原则）

**代码逻辑**:
- [ ] 边界条件处理
- [ ] 错误处理完整
- [ ] 无死代码
- [ ] 性能优化

**安全性**:
- [ ] 输入验证
- [ ] SQL 注入防护
- [ ] XSS 防护
- [ ] 敏感信息不硬编码

**可维护性**:
- [ ] 函数单一职责
- [ ] 模块低耦合
- [ ] 配置与代码分离
- [ ] 日志完整

### 测试流程

1. **测试用例编写** (测试) - 基于 PRD 文档
2. **功能测试** (测试) - 执行测试用例
3. **回归测试** (测试) - 自动化测试脚本
4. **性能测试** (测试) - 压力测试、性能分析
5. **Bug 修复循环** (测试 → 开发 → 测试)
6. **发布验证** (PM + 测试) - 发布检查清单

**通过标准**:
- 测试用例 100% 通过
- 无 P0/P1 级别 Bug
- 性能指标达标

## 📊 任务管理规范

### 任务分解原则

- **粒度**: 每个任务 ≤ 4 小时
- **独立性**: 每个任务可独立测试
- **交付物**: 每个任务有明确交付物
- **责任人**: 每个任务有明确负责人

### 任务卡片模板

```markdown
## 任务卡片 [ID]

**任务**: [任务描述]  
**角色**: [负责角色]  
**优先级**: P0/P1/P2  
**工时**: [预计小时]  
**截止**: [日期时间]  

**输入**:
- [前置任务交付物]

**工作内容**:
1. [工作项 1]
2. [工作项 2]

**交付物**:
- [交付物 1]
- [交付物 2]

**验收标准**:
- [标准 1]
- [标准 2]
```

### 进度追踪

| 状态 | 说明 | 行动 |
|------|------|------|
| 🟢 正常 | 按计划进行 | 继续 |
| 🟡 风险 | 预计超时 <2 天 | 关注 + 补救措施 |
| 🔴 超时 | 已过截止日期 | 立即处理 + 调整期限 |
| ✅ 完成 | 已交付 + 验收通过 | 归档 |
| ⏸️ 阻塞 | 等待外部输入 | 升级 + 协调 |

## 📢 汇报机制

### 每日站会 (08:00 / 20:00)

**内容**:
- 昨日完成
- 今日计划
- 阻塞问题

### 任务开始汇报 (实时)

**内容**:
- 任务理解
- 预计时间
- 潜在风险

### 任务完成汇报 (实时)

**内容**:
- 交付物
- 测试结果
- 下一步计划

### 风险预警 (实时)

**触发条件**:
- 预计超时 <2 天
- 技术难点无法解决
- 依赖任务延迟

**内容**:
- 风险描述
- 影响分析
- 建议措施

### 阶段评审 (每周)

**内容**:
- 阶段总结
- 质量报告
- 下周计划
