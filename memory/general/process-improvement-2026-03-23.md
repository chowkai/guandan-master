# 流程改进记录 - 2026-03-23

**创建日期**: 2026-03-23  
**触发事件**: 掼蛋 Phase 2 待确认事项未及时上报  
**改进目标**: 优化决策事项上报流程，确保主动汇报、不等待询问

---

## 🔍 问题描述

**事件**: 掼蛋 Phase 2 完成后，有 4 个待确认事项（云服务商/域名备案/P2P 策略/UI 主色调）没有主动上报，等待用户询问才暴露。

**影响**:
- 用户需要主动询问才发现问题
- 违背了「第二原则：凡事有交代」
- 可能延误 Phase 3 开发进度

---

## 🎯 根因分析

### 原因 1：任务文档没有明确"待确认事项上报"任务
- `PHASE2-TASKS.md` 中只有 4 个执行任务
- 没有"待确认事项收集与上报"这个独立任务
- Agent 完成任务后只汇报交付物，没有汇报"待决策事项"

### 原因 2：PM 没有在执行过程中主动收集待确认事项
- task-004（完成汇报）是最后才执行的
- 待确认事项在 task-001 和 task-003 完成时就已经产生了
- 但没有在产生时立即上报，而是堆积到最后

### 原因 3：CORE.md 中的"主动汇报机制"定义不清晰
- 只写了"阻塞求助（需要决策）"
- 但没有定义什么是"需要决策"的事项
- Agent 无法判断哪些事项需要上报

### 原因 4：缺少决策事项追踪工具
- 没有统一的 `decision-log.md` 来追踪待决策事项
- PM 无法一目了然地看到哪些事项等待决策
- 无法主动提醒用户决策

---

## ✅ 改进措施

### 改进 1：创建决策追踪文件

**文件**: `tasks/decision-log.md`

**内容**:
- 决策事项总览表（序号/事项/优先级/状态/截止时间/影响）
- 每个事项的详细信息（选项/建议/理由/影响）
- 主动汇报记录
- 更新历史

**维护规则**:
1. 新任务完成后 10 分钟内必须检查并录入待确认事项
2. 决策事项录入后立即上报用户（不等待询问）
3. 到期前 24 小时主动提醒
4. 决策完成后更新状态并归档

---

### 改进 2：更新 CORE.md 增加决策事项追踪机制

**新增章节**:
```markdown
### 决策事项追踪机制

**PM 必须维护**: `tasks/decision-log.md`

**PM 职责**:
- 每个任务完成后 10 分钟内检查并录入待确认事项
- 决策事项录入后立即上报用户（不等待询问）
- 到期前 24 小时主动提醒
- 决策完成后更新状态并归档

**任务负责人职责**:
- 任务完成后检查：是否有未决的技术决策/产品选择题/风险事项？
- 有则按格式上报 PM：`【待确认事项】task-XXX + 事项详情`
- 无则汇报"无待确认事项"
```

---

### 改进 3：更新 HEARTBEAT.md 增加决策事项检查

**新增章节**:
```markdown
# 决策事项检查（每次 Heartbeat 必做）
- [ ] 读取 `tasks/decision-log.md`
- [ ] 检查是否有决策事项即将到期（<24 小时）
- [ ] 有到期风险则立即提醒用户
- [ ] 更新决策事项状态
```

```markdown

  - 事项 2：...

**待确认事项检查清单**（任务完成后必填）:
- [ ] 本任务是否有未决的技术决策？
- [ ] 是否有影响后续任务的依赖事项？
- [ ] 是否有产品方向的选择题？
- [ ] 是否有风险需要用户知悉？
```

## 📝 经验教训

### 学到的教训
1. **流程不能依赖记忆** - 必须有明确的文档和检查清单
2. **主动汇报需要工具支持** - `decision-log.md` 是关键
3. **Heartbeat 是主动汇报的好时机** - 定期检查防止遗漏
4. **PM 职责要明确** - 决策追踪是 PM 的核心职责之一

### 可复用的改进
- `decision-log.md` 可用于所有项目
- 待确认事项检查清单可纳入任务模板
- Heartbeat 决策检查机制可推广
